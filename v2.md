# EndpointGraph V2 Plan

V1 shipped a working proof of concept: login with GitHub, paste a repo URL, get a dependency graph.
V2 makes it a real product — per-user data, a proper repo browser, deeper analysis, and security at the DB layer.

---

## What changes and why

### 1. Per-user + per-repo data isolation + RLS

**V1 problem:** Every user's analyses land in the same DB tables. Analyzing two repos mixes their endpoints into one graph. User A can see User B's services.

**V2 fix:** Add both `user_id` and `repo_id` to `services`. `user_id` scopes data to the logged-in user. `repo_id` (the GitHub full repo name, e.g. `iamaryan07/sample-services`) scopes the graph view to a single repo — so analyzing repo A and repo B produces two separate graphs, not one merged one. Enable Row Level Security (RLS) on all three tables so Supabase enforces user isolation at the DB layer.

Schema changes:
```sql
ALTER TABLE public.services
  ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  ADD COLUMN repo_id VARCHAR(255),           -- e.g. "iamaryan07/sample-services"
  ADD COLUMN last_analyzed_at TIMESTAMP DEFAULT NOW();

-- Unique constraint: same service name in same repo for same user = one row
ALTER TABLE public.services
  ADD CONSTRAINT services_user_repo_name_unique UNIQUE (user_id, repo_id, name);

-- RLS policies
ALTER TABLE public.services ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.endpoints ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.consumer_edges ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users see own services"
  ON public.services FOR ALL
  USING (user_id = auth.uid());

CREATE POLICY "users see own endpoints"
  ON public.endpoints FOR ALL
  USING (
    service_id IN (SELECT id FROM public.services WHERE user_id = auth.uid())
  );

CREATE POLICY "users see own edges"
  ON public.consumer_edges FOR ALL
  USING (
    caller_service_id IN (SELECT id FROM public.services WHERE user_id = auth.uid())
  );
```

FastAPI extracts `user_id` by calling `GET https://api.github.com/user` with the provider token — the GitHub user ID maps to the Supabase `auth.users` row created on OAuth login.

**Re-analysis must upsert, not insert.** Services and endpoints use `ON CONFLICT ... DO UPDATE` so re-analyzing the same repo updates existing rows instead of creating duplicates. The `consumer_edges` upsert already exists in V1 — services and endpoints need the same treatment.

```sql
-- Upsert service (on re-analysis, update last_analyzed_at)
INSERT INTO services (name, language, repo_url, user_id, repo_id, last_analyzed_at)
VALUES ($1, $2, $3, $4, $5, NOW())
ON CONFLICT (user_id, repo_id, name)
DO UPDATE SET last_analyzed_at = NOW(), language = EXCLUDED.language;

-- Upsert endpoint (on re-analysis, keep existing row)
INSERT INTO endpoints (service_id, method, path, spec_source)
VALUES ($1, $2, $3, $4)
ON CONFLICT (service_id, method, path)
DO NOTHING;
```

### 2. Graph persistence on page load

**V1 problem:** The graph only appears after the user clicks Analyze. Refreshing the page wipes it. There is no memory of what was last viewed.

**V2 fix:** On `/graph?repo=iamaryan07/sample-services`, fetch the graph for that repo on mount via `useEffect`. The URL carries the repo context so refresh always restores the right graph.

```js
useEffect(() => {
  if (!repoId) return
  fetchGraph(repoId).then(/* render nodes/edges */)
}, [repoId])
```

`GET /graph` becomes `GET /graph?repo_id=iamaryan07/sample-services` — filtered to one repo + one user.

### 3. Repo browser — no more pasting URLs

**V1 problem:** Users paste a raw GitHub URL. No way to know which repos are already tracked.

**V2 fix:** Replace the URL input with a repo browser. After login, `/repos` calls `GET https://api.github.com/user/repos` using the provider token and lists all the user's repos. Each repo has a **Track** button. Already-tracked repos show a **Re-analyze** button and a link to their graph.

New pages:
```
/repos           ← repo list (replaces the URL input box)
/graph?repo=...  ← graph view, scoped to one repo
```

New frontend API call in `lib/api.js`:
```js
export async function fetchUserRepos() {
  const token = await getGitHubToken()
  const res = await fetch('https://api.github.com/user/repos?per_page=100&sort=pushed', {
    headers: { Authorization: `Bearer ${token}` }
  })
  return res.json()  // array of { name, full_name, private, updated_at }
}
```

No arbitrary public URL input. The user can only track repos the GitHub token can see.

### 4. Recursive service discovery — full depth, no limit

**V1 problem:** Only scans top-level directories. Services nested at any depth are silently ignored.

**V2 fix:** Walk the full directory tree with no depth cap. Skip well-known non-service directories. Once a folder is identified as a service (contains `openapi.yaml/json` or source files), stop descending into it — a service folder is a leaf.

```python
IGNORED_DIRS = {'.git', 'node_modules', '.venv', '__pycache__', 'dist', 'build', '.next', 'coverage'}

def find_service_folders(root: str) -> list[str]:
    results = []
    def walk(path):
        for entry in os.listdir(path):
            full = os.path.join(path, entry)
            if not os.path.isdir(full) or entry in IGNORED_DIRS:
                continue
            if _is_service_folder(full):
                results.append(full)
            else:
                walk(full)
    walk(root)
    return results
```

No hardcoded depth limit — the ignored dirs set is what bounds the traversal.

### 5. JavaScript / TypeScript support

**V1 problem:** tree-sitter analysis only runs on `.py` files. A JS/TS service is invisible.

**V2 fix:** Extend `code_parser.py` to parse `.js`, `.jsx`, `.ts`, and `.tsx` files. Two jobs, same as Python:

**Job 1 — Route decorators (what a service exposes):**
- Express: `app.get('/path', ...)`, `router.post('/path', ...)`
- Next.js API routes: filename-based routing (`/app/api/users/route.js` → `GET /api/users`)

**Job 2 — HTTP call sites (what a service calls):**
- `fetch('http://service/path')`
- `axios.get('http://service/path')`

The URL matcher operates on the extracted URL string — it works regardless of source language.

`endpoints.spec_source` values in V2:
| Value | Meaning |
|---|---|
| `openapi` | Found in openapi.yaml |
| `decorator` | Python route decorator |
| `decorator_js` | JS/TS Express route handler |
| `nextjs_route` | Next.js file-based routing |

## New API routes

| Method | Route | What it does |
|---|---|---|
| GET | `/repos` | Calls GitHub API, returns user's repos with `tracked: bool` for each |
| GET | `/graph?repo_id=...` | Graph filtered to one repo + one user |
| DELETE | `/services/{id}` | Remove a tracked repo and all its data |

## What stays the same

- FastAPI backend, asyncpg, Supabase
- React Flow graph visualization
- Impact panel on node click
- Search bar
- No Docker, no background jobs
- `source = 'static'` only (log ingestion is still V3+)

## What's explicitly out of V2

- gRPC / .proto parsing
- GitHub PR comment bot ("this PR breaks 3 consumers")
- Field-level impact analysis
- Log ingestion (Envoy, Istio)
- Teams / org-level sharing
- Deprecation tracking
