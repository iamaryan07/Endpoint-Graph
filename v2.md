# EndpointGraph V2 Plan

V1 shipped a working proof of concept: login with GitHub, paste a repo URL, get a dependency graph.
V2 makes it a real product — per-user data, a proper repo browser, deeper analysis, and security at the DB layer.

---

## What changes and why

### 1. Per-user data isolation + RLS

**V1 problem:** Every user's analyses land in the same DB tables. User A can see User B's services.

**V2 fix:** Add a `user_id` column (referencing `auth.users.id`) to `services`. All downstream tables (`endpoints`, `consumer_edges`) cascade from `services`, so filtering by `user_id` on `services` is enough. Enable Row Level Security (RLS) on all three tables so Supabase enforces isolation at the DB layer — not just at the application layer.

Schema changes:
```sql
ALTER TABLE public.services ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE;

-- RLS policies
ALTER TABLE public.services ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.endpoints ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.consumer_edges ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users see own services"
  ON public.services FOR ALL
  USING (user_id = auth.uid());

CREATE POLICY "users see own endpoints"
  ON public.endpoints FOR ALL
  USING (
    service_id IN (SELECT id FROM public.services WHERE user_id = auth.uid())
  );

CREATE POLICY "users see own edges"
  ON public.consumer_edges FOR ALL
  USING (
    caller_service_id IN (SELECT id FROM public.services WHERE user_id = auth.uid())
  );
```

FastAPI extracts `user_id` from the GitHub token by calling `GET https://api.github.com/user` — the GitHub user ID maps to the Supabase `auth.users` row created on OAuth login.

---

### 2. Repo browser — no more pasting URLs

**V1 problem:** Users paste a raw GitHub URL. They have to know the exact URL, it's error-prone, and there's no way to know which repos have already been tracked.

**V2 fix:** Replace the URL input with a repo browser. After login, the `/repos` page calls `GET https://api.github.com/user/repos` using the provider token and shows all of the user's repos. Each repo has a **Track** button. Clicking it runs analysis on that repo. Previously tracked repos show a **Re-analyze** button and a link to their graph.

New pages:
```
/repos       ← repo list (replaces the URL input box)
/graph       ← graph view, now scoped to one repo at a time
```

New frontend API call in `lib/api.js`:
```js
export async function fetchUserRepos() {
  const token = await getGitHubToken()
  const res = await fetch('https://api.github.com/user/repos?per_page=100&sort=pushed', {
    headers: { Authorization: `Bearer ${token}` }
  })
  return res.json()  // array of { name, full_name, private, updated_at }
}
```

No arbitrary public URL input. The user can only track their own repos (what the GitHub token can see). This removes the attack surface of users cloning arbitrary public repos through the backend.

### 3. Recursive service discovery

**V1 problem:** Only scans the immediate top-level directories of a cloned repo. Any repo where services are nested (e.g. `backend/services/user-service/`) is silently ignored.

**V2 fix:** Walk the directory tree up to a configurable depth (default: 3 levels). Collect every folder that contains an `openapi.yaml/json` or `.py`/`.js` files directly inside it. Stop descending into `node_modules`, `.git`, `.venv`, `__pycache__`, `dist`, `build`.

```python
IGNORED_DIRS = {'.git', 'node_modules', '.venv', '__pycache__', 'dist', 'build', '.next'}

def find_service_folders(root: str, max_depth: int = 3) -> list[str]:
    results = []
    def walk(path, depth):
        if depth > max_depth:
            return
        for entry in os.listdir(path):
            full = os.path.join(path, entry)
            if not os.path.isdir(full) or entry in IGNORED_DIRS:
                continue
            if _is_service_folder(full):
                results.append(full)
            else:
                walk(full, depth + 1)
    walk(root, 0)
    return results
```

Note: once a folder is identified as a service, its subdirectories are not recursed into — a service folder is a leaf.

### 4. JavaScript support

**V1 problem:** tree-sitter analysis only runs on `.py` files. A JS/TS service is invisible — it shows up as a service node with no endpoints and no edges.

**V2 fix:** Extend `code_parser.py` to also parse `.js` and `.ts` files. Two jobs, same as Python:

**Job 1 — Route decorators (what a service exposes):**
- Express: `app.get('/path', ...)`, `router.post('/path', ...)`
- Next.js API routes: filename-based routing (`/app/api/users/route.js` → `GET /api/users`)

**Job 2 — HTTP call sites (what a service calls):**
- `fetch('http://service/path')`
- `axios.get('http://service/path')`

The same URL matcher works regardless of language — it operates on the extracted URL string, not the language.

`spec_source` for JS routes discovered via decorator: `"decorator_js"`.

`endpoints.spec_source` values in V2:
| Value | Meaning |
|---|---|
| `openapi` | Found in openapi.yaml |
| `decorator` | Python route decorator |
| `decorator_js` | JS/Express route handler |
| `nextjs_route` | Next.js file-based routing |

```sql
-- Track which repo each service came from (already exists as repo_url, but scope it to user)
-- No new columns needed beyond user_id on services
-- One addition: track when a repo was last analyzed
ALTER TABLE public.services ADD COLUMN last_analyzed_at TIMESTAMP DEFAULT NOW();
```

## New API routes

| Method | Route | What it does |
|---|---|---|
| GET | `/repos` | Calls GitHub API, returns user's repos with `tracked: bool` for each |
| GET | `/graph?repo_url=...` | Existing `/graph` but filtered to one repo + one user |
| DELETE | `/services/{id}` | Remove a tracked repo and all its data |

## What stays the same

- FastAPI backend, asyncpg, Supabase
- React Flow graph visualization
- Impact panel on node click
- Search bar
- No Docker, no background jobs
- `source = 'static'` only (log ingestion is still V3+)

---

## What's explicitly out of V2

- gRPC / .proto parsing
- GitHub PR comment bot ("this PR breaks 3 consumers")
- Field-level impact analysis
- Log ingestion (Envoy, Istio)
- Teams / org-level sharing
- Deprecation tracking
